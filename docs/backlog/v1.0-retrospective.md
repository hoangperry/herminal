# v1.0.0 retrospective — herminal leaves beta

**Date:** 2026-06-06
**Tag:** v1.0.0 (build 22)

herminal reaches 1.0. This is a milestone tag, not a rewrite: it
declares that everything in the PRD's v1 scope is built, reviewed, and
guarded, and that the surface is stable enough to stand behind.

## What 1.0 means here

- **Every PRD v1 capability shipped.** Native libghostty terminal core,
  Vietnamese Telex/VNI IME, the agent dashboard, per-session SQLite
  notes, the SSH manager, premium dark + light chrome, telemetry-free
  crash diary, Developer-ID sign + notarize pipeline.
- **Post-MVP themes closed.** Sessions (Theme H — Claude browser,
  restore, named workspaces, live cwd, opt-in re-run), workspace
  ergonomics (Theme C — recursive split trees, directional focus,
  focused-pane outline, cwd-inheriting tabs, font size, pane zoom),
  agent depth (Theme A), light theme + drag-resize (Theme C), distribution
  (Theme E).
- **The last "every terminal has this" verbs** (v1.0): live font size
  (⌘+/−/0) and pane zoom (⌘⇧Return).

## Quality bar at 1.0

- **143 unit tests** (Swift Testing) + a **9-check dogfood gate** of live
  GUI integration guards (smoke, baseline injection, codex/ssh/compat,
  clipboard, title, session-restore, pane-nav), all green. The gate
  retries a flaky check once so it's reliable enough to gate releases.
- **Five parallel code + security review passes** (M11, M13, C2, v0.5,
  v1.0). Every pass found a real issue — the most serious a boot-loop DoS
  (deeply-nested `workspace.json` overflowing the JSON decoder, v0.5),
  all fixed + regression-tested. Full log: `docs/REVIEW.md`.
- **0 crashes** in dogfood/stress; ~8,700 Swift LOC; 2 external deps
  (SQLite.swift, libghostty).

## What 1.0 deliberately is NOT

Per the PRD's won't-ship list: no Linux/Windows, no cloud sync/accounts,
no plugin or theme marketplace, no in-terminal AI chat, no App Store
build, no telemetry. These aren't gaps — they're the product's spine.

## Deferred past 1.0 (feedback-gated)

- SSH manager search / groups / per-host keypair UI — gated on real
  host-count + a security review.
- Recursive-split image rendering, more agent kinds, bilingual site.
- Diary opt-in upload + daily rotation — wait for the complaint.

## Owner-gated for the actual release

The code is at 1.0.0 on `main`, the dogfood gate is green, and
`Scripts/release.sh 1.0.0` will run the gate → build → sign → notarize →
tag → zip. Two steps remain that only the owner can do because they
require secrets:

1. **Notarize + publish** — needs the Developer-ID identity (present in
   the keychain) **and** the notarytool credentials/profile (not on this
   machine / not discoverable without the owner). Then `git push` the tag
   + `gh release create` + bump the Homebrew cask sha256.
2. **Sparkle auto-update** — needs the owner's EdDSA key (a secret). Full
   handoff in `docs/SPARKLE-NEXT.md`.

Until then, 8 versions of work (v0.4.3 → v1.0.0) sit released-ready but
unpublished; the public release is still v0.4.2.

## The honest takeaway

The engineering is done and verified. The remaining distance to a
*published* 1.0 isn't code — it's the owner running the notarize step
with credentials that must never enter the repo. Everything that can be
delivered to git autonomously, has been.
